# Fuzzing Coverage Workflow - Input/Output Specification

This document provides a detailed specification of inputs and outputs for each step in the Fuzzing Coverage Workflow.

## Step 1: Extract Target Functions

**Type:** Task (PROGRAM)

**Description:** Extract all targeted functions from the fuzzing suite using the extract_target_functions.py script

**Inputs:**
- Directory: `test/recon` (fuzzing suite directory)
- Format: Directory containing fuzzing test files

**Command:**
```bash
extract-target-functions --targets test/recon --return-json
```

**Outputs:**
- File: `magic/target-functions.json`
- Format: JSON containing list of target functions extracted from the fuzzing suite
- Captured: Yes

---

## Step 2: Build Artifacts

**Type:** Task (PROGRAM)

**Description:** Generate Forge build artifacts with build-info

**Inputs:**
- Source: Solidity contracts in the project
- Format: `.sol` files in the project structure

**Command:**
```bash
forge clean && forge build --build-info
```

**Outputs:**
- Directory: `out/build-info/`
- Format: JSON build artifacts generated by Forge containing contract compilation metadata
- Captured: No

---

## Step 3a: Filter Build Artifacts

**Type:** Task (PROGRAM)

**Description:** Filter build-info to remove sources without AST data (foundry-pp files)

**Inputs:**
- File: First non-filtered JSON file in `out/build-info` directory
- Format: Forge build artifact JSON file

**Command:**
```bash
filter-build-info $(find out/build-info -name '*.json' -not -name '*.filtered.json' | head -1)
```

**Tool:** `filter-build-info` (CLI tool from `tools/filter_build_info`)

**Purpose:**
Foundry generates internal preprocessor files (in `foundry-pp/` directory) that lack AST data. These files cause `sol-expand` to fail with "Unable to detect reader configuration" errors. This step filters the build-info JSON to remove such sources before passing to `sol-expand`.

**What Gets Filtered:**
- `foundry-pp/DeployHelper*.sol` files (Foundry's internal preprocessing artifacts)
- Any other source files without valid AST data

**Outputs:**
- File: `out/build-info/*.filtered.json` (filtered build artifact)
- Format: Forge build artifact JSON file with sources lacking AST data removed
- Captured: No

---

## Step 3b: Extract Context with sol-expand

**Type:** Task (PROGRAM)

**Description:** Run sol-expand tool to extract context from filtered build artifacts

**Inputs:**
- File: Filtered JSON file in `out/build-info` directory (output from Step 3a)
- Format: Filtered Forge build artifact JSON file

**Command:**
```bash
sol-expand --extract-context $(find out/build-info -name '*.filtered.json' | head -1)
```

**Outputs:**
- Directory: `context_output/`
- Format: Extracted contract context and metadata from sol-expand tool
- Captured: No

**Note:** This step now uses the filtered build-info from Step 3a to avoid AST parsing errors.

---

## Step 4: Identifying Touched Functions

**Type:** Task (PROGRAM)

**Description:** Identify all contracts that are touched by the contracts targeted in the system that we care about

**Inputs:**
- Directory: `context_output/` (from sol-expand)
- File: `magic/target-functions.json` (from Step 1)
- Format:
  - `context_output/`: Sol-expand extracted context
  - `target-functions.json`: JSON list of target functions

**Command:**
```bash
touched-function-identifier --sol-expand-dir context_output --target-functions magic/target-functions.json --return-json
```

**Outputs:**
- File: `magic/functions-to-cover.json`
- Format: JSON containing list of all functions that need to be covered based on touched contracts
- Captured: Yes

---

## Step 5: Identifying Meaningful Values

**Type:** Task (OPENCODE - Agent)

**Description:** Identify values that are useful for clamping target function handlers with

**Inputs:**
- Agent Definition: `./.opencode/agent/coverage-phase-1.md`
- Implicit Inputs:
  - `magic/functions-to-cover.json` (from Step 4)
  - Contract source code
  - Build artifacts
- Format: Agent reads from JSON files and contract source

**Agent:** `coverage-phase-1`

**Outputs:**
- Variable: Implementation-dependent (determined by agent)
- Format: Identified meaningful values for clamping handlers (may include constants, bounds, special values)
- Captured: Dependent on agent implementation

---

## Step 6: Creating Clamped Handlers

**Type:** Task (OPENCODE - Agent)

**Description:** Implement clamped function handlers to achieve full coverage on the system setup

**Inputs:**
- Agent Definition: `./.opencode/agent/coverage-phase-2.md`
- Implicit Inputs:
  - `magic/functions-to-cover.json` (from Step 4)
  - Meaningful values identified in Step 5
  - Existing fuzzing test files
- Format: Agent reads JSON files and modifies test contracts

**Agent:** `coverage-phase-2`

**Outputs:**
- Files: Modified or new handler functions in test contracts
- Format: Solidity code with clamped handlers
- Captured: No (code changes committed)

---

## Step 7: Creating Shortcut Handlers

**Type:** Task (OPENCODE - Agent)

**Description:** Implement shortcut handlers that allow the fuzzer to reach coverage faster

**Inputs:**
- Agent Definition: `./.opencode/agent/coverage-phase-3.md`
- Implicit Inputs:
  - `magic/functions-to-cover.json` (from Step 4)
  - Existing clamped handlers (from Step 6)
  - Contract source code
- Format: Agent reads JSON files and contract code

**Agent:** `coverage-phase-3`

**Outputs:**
- Files: Modified or new shortcut handler functions in test contracts
- Format: Solidity code with shortcut handlers
- Captured: No (code changes committed)

---

## Step 8: Run Echidna Programmatically

**Type:** Task (PROGRAM)

**Description:** Run Echidna programmatically

**Inputs:**
- Files: All test contracts with handlers (from Steps 6-7)
- Config: `echidna.yaml`
- Contract: `CryticTester`
- Format: Solidity test contracts and YAML configuration

**Command:**
```bash
echidna . --contract CryticTester --config echidna.yaml --format text --timeout 1800 --test-limit 99999999999999999999 --disable-slither --test-mode exploration
```

**Outputs:**
- Directory: `echidna/` (corpus and coverage data)
- Format: Echidna output files including:
  - Coverage reports
  - Test corpus
  - Execution logs
- Captured: No (files generated in echidna directory)

---

## Step 9: Evaluate Coverage

**Type:** Task (PROGRAM)

**Description:** Run coverage evaluation tool

**Inputs:**
- Directory: `magic/` (contains functions-to-cover.json)
- Directory: `echidna/` (contains coverage data from Step 8)
- Format:
  - JSON files in magic directory
  - Echidna coverage output

**Command:**
```bash
covg-eval magic/ echidna/ --return-json
```

**Outputs:**
- File: `magic/functions-missing-covg-{timestamp}.json`
- Format: JSON containing list of functions that were not covered by fuzzing
- Captured: Yes
- Note: File is only created if there are missing coverage functions

---

## Step 10: Initial Check of Coverage

**Type:** Decision (FILE_EXISTS)

**Description:** Determine if coverage needs to be improved

**Inputs:**
- Pattern: `functions-missing-covg-*.json` in magic directory
- Format: Checks for existence of coverage gap file

**Decision Logic:**
- If file exists (value = 1): Jump to "Step 11: Improving Coverage"
- If file does not exist (value = 0): Jump to "Step 15: Workflow Complete"

**Outputs:**
- Control Flow: Determines next step execution
- Format: Workflow branch decision

---

## Step 11: Improving Coverage

**Type:** Task (OPENCODE - Agent)

**Description:** Analyze coverage gaps and improve clamped handlers to achieve better coverage

**Inputs:**
- Agent Definition: `./.opencode/agent/coverage-phase-5.md`
- File: Latest `magic/functions-missing-covg-{timestamp}.json` (from Step 9)
- Implicit Inputs:
  - Existing handler implementations
  - Contract source code
  - `magic/functions-to-cover.json`
- Format: JSON coverage gap data and Solidity code

**Agent:** `coverage-phase-5`

**Outputs:**
- Files: Modified or new handler functions to address coverage gaps
- Format: Solidity code with improved handlers
- Captured: No (code changes committed)

---

## Step 12: Run Echidna Programmatically (Iteration)

**Type:** Task (PROGRAM)

**Description:** Run Echidna programmatically (same as Step 8, but after coverage improvements)

**Inputs:**
- Files: Updated test contracts with improved handlers (from Step 11)
- Config: `echidna.yaml`
- Contract: `CryticTester`
- Format: Solidity test contracts and YAML configuration

**Command:**
```bash
echidna . --contract CryticTester --config echidna.yaml --format text --timeout 1800 --test-limit 99999999999999999999 --disable-slither --test-mode exploration
```

**Outputs:**
- Directory: `echidna/` (updated corpus and coverage data)
- Format: Echidna output files including:
  - Updated coverage reports
  - Updated test corpus
  - Execution logs
- Captured: No (files generated in echidna directory)

---

## Step 13: Evaluate Coverage (Iteration)

**Type:** Task (PROGRAM)

**Description:** Run coverage evaluation tool (same as Step 9, but after improvements)

**Inputs:**
- Directory: `magic/` (contains functions-to-cover.json)
- Directory: `echidna/` (contains updated coverage data from Step 12)
- Format:
  - JSON files in magic directory
  - Echidna coverage output

**Command:**
```bash
covg-eval magic/ echidna/ --return-json
```

**Outputs:**
- File: `magic/functions-missing-covg-{timestamp}.json` (new timestamp)
- Format: JSON containing remaining functions not covered by fuzzing
- Captured: Yes
- Note: File is only created if there are still missing coverage functions

---

## Step 14: Coverage Improvement Decision Check

**Type:** Decision (FILE_EXISTS)

**Description:** Check if coverage needs to be improved

**Inputs:**
- Pattern: `functions-missing-covg-*.json` in magic directory
- Format: Checks for existence of coverage gap file

**Decision Logic:**
- If file exists (value = 1): Jump to "Step 11: Improving Coverage" (loop back)
- If file does not exist (value = 0): Continue to next step

**Outputs:**
- Control Flow: Determines next step execution (loop or continue)
- Format: Workflow branch decision

---

## Step 15: Workflow Complete

**Type:** Task (PROGRAM)

**Description:** Workflow has completed successfully - full coverage achieved or no coverage gaps detected

**Inputs:**
- None (termination step)

**Command:**
```bash
echo 'Fuzzing coverage workflow completed successfully'
```

**Outputs:**
- Console: Success message
- Format: Text output
- Captured: No

---

## Workflow Flow Summary

```
Step 1 (Extract Target Functions)
  ↓ [magic/target-functions.json]
Step 2 (Build Artifacts)
  ↓ [out/build-info/*.json]
Step 3a (Filter Build Artifacts)
  ↓ [out/build-info/*.filtered.json]
Step 3b (Extract Context)
  ↓ [context_output/]
Step 4 (Identify Touched Functions)
  ↓ [magic/functions-to-cover.json]
Step 5 (Identify Meaningful Values)
  ↓ [meaningful values]
Step 6 (Create Clamped Handlers)
  ↓ [handler code]
Step 7 (Create Shortcut Handlers)
  ↓ [shortcut handler code]
Step 8 (Run Echidna)
  ↓ [echidna/ coverage data]
Step 9 (Evaluate Coverage)
  ↓ [magic/functions-missing-covg-{timestamp}.json]
Step 10 (Decision)
  ├─ Coverage gaps exist → Step 11
  └─ No gaps → Step 15
Step 11 (Improve Coverage)
  ↓ [improved handler code]
Step 12 (Run Echidna - Iteration)
  ↓ [updated echidna/ coverage data]
Step 13 (Evaluate Coverage - Iteration)
  ↓ [magic/functions-missing-covg-{timestamp}.json]
Step 14 (Decision)
  ├─ Coverage gaps still exist → Step 11 (loop)
  └─ No gaps → Continue
Step 15 (Workflow Complete)
```

## Key File Formats

### magic/target-functions.json
```json
{
  "target_functions": [
    {
      "contract": "ContractName",
      "function": "functionName",
      "signature": "functionName(type1,type2)"
    }
  ]
}
```

### magic/functions-to-cover.json
```json
{
  "functions_to_cover": [
    {
      "contract": "ContractName",
      "function": "functionName",
      "signature": "functionName(type1,type2)",
      "source_file": "path/to/contract.sol"
    }
  ]
}
```

### magic/functions-missing-covg-{timestamp}.json
```json
{
  "missing_coverage": [
    {
      "contract": "ContractName",
      "function": "functionName",
      "signature": "functionName(type1,type2)",
      "reason": "description of why coverage was not achieved"
    }
  ],
  "timestamp": "2025-12-04T10:30:00Z"
}
```

## Notes

- Steps with `shouldCommitChanges: true` will commit their changes to git
- Steps with `shouldCreateSummary: true` will generate execution summaries
- The workflow includes a coverage improvement loop (Steps 11-14) that continues until full coverage is achieved
- Timestamps in filenames are generated dynamically at runtime
- Agent-based steps (OPENCODE type) have flexible outputs defined by their agent definitions

## Tools

### filter-build-info

**Location:** `tools/filter_build_info/`

**Installation:**
```bash
uv tool install --editable ./tools/filter_build_info
```

**Purpose:** Filters Forge build-info JSON files to remove sources without valid AST data, preventing sol-expand from failing on Foundry's internal preprocessor files.

**Usage:**
```bash
# Basic usage
filter-build-info out/build-info/abc123.json

# Specify output
filter-build-info input.json --output filtered.json

# Verbose mode
filter-build-info input.json --verbose
```

**Common Issues Addressed:**
- Fixes: `Error: Unable to detect reader configuration for entry "foundry-pp/DeployHelper*.sol"`
- Removes: 10+ foundry-pp preprocessor files that lack AST data
- Ensures: sol-expand can process the build artifacts without errors

**Output:** Creates a `.filtered.json` file with problematic sources removed, outputting the path to stdout for easy use in pipelines.

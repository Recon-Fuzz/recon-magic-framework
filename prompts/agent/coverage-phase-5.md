## Role

You will use the `analysis` field in `functions-missing-covg-N.json` to determine the appropriate solution: clamped handlers for simple parameter issues or shortcut functions for complex state setup scenarios.

## Step 1 - Finding Missing Coverage with Analysis
The agents before you will have run the fuzzer, evaluated coverage, and analyzed the gaps. You should read `functions-missing-covg-N.json`, choosing the one that has the largest value for N if there are multiple.

This file is generated by `covg-eval` and enhanced with analysis by `coverage-phase-3`. It has the following structure:

The `missing_coverage` array contains entries with:
- Function and contract information
- Uncovered code snippets with the last covered line

## Step 2 - Use Analysis to Guide Implementation

For each entry in the `missing_coverage` array:
1. **Read the `analysis` field** - This tells you:
   - What the root cause of the blockage is
   - What type of fix is recommended (clamped handler, new target function, state initialization)
   - Which parameters need clamping or what state needs to be set up

2. **Optionally cross-reference with `merged-paths-prerequisites.json`**:
   - If the analysis mentions prerequisites or complex state setup
   - Check `paths` array for matching execution path conditions
   - Review `prerequisite_functions` to understand the full call sequence

3. **Determine the implementation approach**:
   - The `analysis` field will guide you toward either:
     - **Clamped handler** - for parameter constraints
     - **Shortcut function** - for multi-step state setup
     - **New target handler** - for missing state-changing functions

## Step 3 - Decision: Clamped Handler vs Shortcut Function

Based on the `analysis` field and cross-reference with `merged-paths-prerequisites.json`, determine the appropriate fix:

### Use Case A: Single Function Call → Implement Clamped Handler
**When to use (look for these in the `analysis` field):**
- Analysis mentions "clamp the parameter space" or "constrain parameter values"
- Analysis identifies specific parameters that need to be authorized/valid
- The uncovered lines can be reached by calling **one function** with specific parameter values
- No complex state setup is required
- The issue is parameter values causing reverts or wrong branches

**Common indicators in analysis:**
- "need to clamp the `onBehalf` parameter to authorized addresses"
- "clamp the `seizedAssets` parameter to match borrower's collateral"
- "fuzzer is passing parameter values that fail validation checks"

**Examples:**
- Authorization check requiring specific address → clamp to actors
- Amount validation requiring specific range → clamp to valid range
- Conditional branch requiring specific value → clamp to trigger branch

### Use Case B: Multiple Function Calls → Implement Shortcut Function
**When to use (look for these in the `analysis` field):**
- Analysis mentions "unreachable state problem" or "state-changing function"
- Analysis mentions need for "multiple prerequisite calls" or "call sequence"
- Analysis describes a scenario requiring several steps to set up state
- The uncovered lines require **multiple prerequisite calls** to set up state
- Matches an execution path in the `paths` array requiring complex setup
- The function needs specific system state that can only be achieved through a sequence

**Common indicators in analysis:**
- "need to call `oracle.setPrice(...)` to initialize the oracle first"
- "requires state-changing calls before the target function can proceed"
- "collateral is modified earlier... need to set up exact state"
- "system state itself is invalid - requires multiple steps"

**Examples:**
- Liquidation requiring: supply → supplyCollateral → borrow → price change
- Withdraw requiring: deposit → accrue interest → time advance
- Complex conditional requiring: initialization → state modification → execution

## Step 4 - Implementation

Based on your decision from Step 3, implement the appropriate solution:

### Solution A: Implement Clamped Handler (Single Function Call)

Follow the rules in `${PROMPTS_DIR}/clamping-handler-rules.md`. **CRITICAL REQUIREMENTS**:
- **Naming**: Clamped handlers MUST always end with `_clamped` suffix
- **Single State Change**: Clamped handlers should NEVER contain more than one state-changing function call - the ONLY state-changing call should be to the unclamped handler
- **Call Hierarchy**: All state-changing calls should be made to handlers (unclamped handlers), NOT directly to contracts
- Apply modulo with `+ 1` (MANDATORY)
- Use values from `meaningful-values.json` when available
- Clamp addresses to actors
- NO require statements or conditional logic

**Example:**
```solidity
function vault_borrow_clamped(MarketParams memory marketParams, uint256 assets, uint256 shares, address receiver) public {
    // Clamp onBehalf to authorized actor
    vault_borrow(marketParams, assets, shares, _getActor(), receiver);
}

function vault_borrow(MarketParams memory marketParams, uint256 assets, uint256 shares, address onBehalf, address receiver) public asActor {
    vault.borrow(marketParams, assets, shares, onBehalf, receiver);
}
```

### Solution B: Implement Shortcut Function (Multiple Function Calls)

Follow the pattern from `coverage-phase-2`. Key points:
- **CRITICAL**: Shortcut functions MUST ONLY be added to the `TargetFunctions` contract - this is a strict requirement
- **NEVER** add shortcut functions to any other contract
- Name: `shortcut_{targetFunc}_{descriptivePathName}` (based on path conditions)
- Call prerequisites using **clamped handlers**
- Read state to determine exact values for triggering branches
- Call target with **unclamped handler** using exact values
- Match one execution path from the `paths` array

**Example:**
```solidity
// Path: "seizedAssets > 0 && position[id][borrower].collateral == 0 && data.length > 0"
// Descriptive name: bySeizedAssets_badDebt_withCallback
function shortcut_liquidate_bySeizedAssets_badDebt_withCallback(
    uint256 supplyAmount,
    uint256 collateralAmount,
    uint256 borrowAmount,
    uint256 newPrice,
    bytes calldata data
) public {
    // Prerequisites - use CLAMPED handlers
    vault_supply_clamped(supplyAmount);
    switchActor(1);
    vault_supplyCollateral_clamped(collateralAmount);
    vault_borrow_clamped(borrowAmount);

    // Make position liquidatable by changing oracle price
    oracle_setPrice(newPrice);

    // Read exact value to trigger "collateral == 0" branch
    Position memory pos = vault.position(defaultMarketId, _getActor());
    uint256 seizedAssets = pos.collateral; // Exact value to zero out collateral

    // Target call - use UNCLAMPED handler to preserve exact value
    switchActor(0);
    vault_liquidate(defaultMarketParams, _getActor(), seizedAssets, 0, data);
}
```

## Step 5 - Mapping Coverage Gaps to Solutions

For each uncovered code block in `functions-missing-covg-N.json`:

1. **Read the source code** to understand the control flow
2. **Check `merged-paths-prerequisites.json`** for matching paths:
   - Look up the target function name
   - Find execution paths that would cover the missing lines
   - Review prerequisite_functions for required state setup
3. **Count required function calls**:
   - If needs 0 prerequisite → **Clamped handler**
   - If needs 1+ prerequisites → **Shortcut function**
4. **Implement the appropriate solution**:
   - Clamped handler: Single function with parameter constraints
   - Shortcut: Multi-step function sequence matching a specific path

## Step 6 - Common Patterns

### Pattern 1: Authorization Blockage
**Missing coverage:** Lines after authorization checks that are failing
**Cross-reference:** Check if target function has authorization-related scenario
**Solution:** **Clamped handler** - constrain parameters to authorized values
**Why:** Single function call, just need valid parameter values

**Common Authorization Patterns:**

1. **Sender Authorization Checks:**
   - `require(_isSenderAuthorized(onBehalf))`
   - `require(msg.sender == owner)`
   - `require(authorized[msg.sender])`
   - **Solution:** clamp address to `_getActor()`

2. **Role-Based Access Control (RBAC):**
   - `require(hasRole(msg.sender, ADMIN_ROLE))`
   - `require(roles[msg.sender] >= MIN_ROLE)`
   - **Solution:** Use `asActor` modifier (actors should have required roles from setup)

3. **Approval/Allowance Checks:**
   - `require(isApproved[owner][spender])`
   - `require(allowance[from][msg.sender] >= amount)`
   - `require(isApprovedForAll(owner, operator))`
   - **Solution:** Clamp address parameters to actors who have pre-approved each other in setup

4. **Ownership Checks:**
   - `require(ownerOf(tokenId) == msg.sender)`
   - `require(balanceOf(address) > 0)`
   - **Solution:** Clamp `tokenId` or address to values owned by `_getActor()`

5. **Delegation Checks:**
   - `require(delegated[from][to])`
   - `require(operators[owner][msg.sender])`
   - **Solution:** Clamp address pairs to pre-established delegate relationships

**Detection Strategy:**
- Examine `last_covered_line` - does it contain an authorization check that passed?
- Look at uncovered lines - is there another authorization check failing?
- Identify which parameter(s) are validated for authorization
- Determine what values would satisfy the check

**Solution Pattern:**
```solidity
function {target}_clamped(...) public {
    // Replace authorization-checked parameter with authorized value
    // Common replacements:
    // - address parameters → _getActor() (authorized actor)
    // - tokenId/index → owned by _getActor()
    // - role/permission → value _getActor() has

    {target}(..., authorizedValue, ...);
}
```

### Pattern 2: Conditional Branch Not Taken
**Missing coverage:** Lines inside conditional blocks that are never executed
**Cross-reference:** Check `paths` array in `merged-paths-prerequisites.json` for matching execution paths
**Solution:** **Shortcut function** (usually) - call prerequisites to set up required state
**Why:** Typically requires multiple steps to create the exact system state needed for the branch

**Common Conditional Patterns:**
- **State variable checks:** `if (balance == 0)`, `if (shares > threshold)`, `if (lastUpdate == 0)`
- **Relational comparisons:** `if (debt > collateral)`, `if (position.amount >= limit)`
- **Boolean flags:** `if (isInitialized)`, `if (hasWithdrawn)`, `if (paused)`
- **Edge case branches:** `if (remaining == 0)`, `if (all assets seized)`, `if (bad debt)`

**Detection Strategy:**
- The `last_covered_line` shows the conditional statement was reached but condition evaluated to false
- Uncovered lines are inside the conditional block
- Branch requires specific system state that can't be achieved with single parameter change

**Solution Approach:**
- Check `paths` array in `merged-paths-prerequisites.json` for an execution path matching this branch
- Identify prerequisites needed to create the required state
- Implement shortcut calling prerequisites + target with exact values to trigger the branch

### Pattern 3: Unreachable State
**Missing coverage:** Lines after validation checks that fail due to uninitialized or invalid system state
**Cross-reference:** Check if prerequisites include required state initialization
**Solution:** **New target handler** (simple state) or **Shortcut function** (complex state)
**Why:** System state itself is invalid - requires state-changing calls before target function can proceed

**Common Unreachable State Patterns:**
- **Uninitialized values:** `require(oracle.price != 0)`, `require(market.created)`, `require(rate != 0)`
- **Missing setup:** `require(tokenExists(id))`, `require(poolInitialized)`, `require(configSet)`
- **Insufficient balances:** `require(reserve > 0)`, `require(liquidity > minLiquidity)`
- **Time-based conditions:** `require(block.timestamp >= unlockTime)`, `require(expired())`

**Detection Strategy:**
- The `last_covered_line` successfully reads a value from storage/state
- Next line's require/check fails because that value is invalid (zero, uninitialized, insufficient)
- No parameter adjustment can fix this - the state itself must change first

**Solution Approach:**
- **Simple state (1 call):** Implement clamped handler that sets the required state value
  - Example: `oracle_setPrice_clamped()` to initialize price
- **Complex state (2+ calls):** Implement shortcut function with full setup sequence
  - Example: shortcut that creates pool, adds liquidity, then executes target
- **New target handler:** If the state-setting function isn't already a target handler, add it

### Pattern 4: Amount/Range Validation
**Missing coverage:** Lines in specific range branches
**Cross-reference:** Check if execution path specifies amount ranges
**Solution:** **Clamped handler** - constrain amounts with modulo
**Why:** Single function call, just need parameter in valid range

## Step 7 - Validation

After implementing fixes:
1. **Verify clamped handlers**:
   - **MUST** have `_clamped` suffix (MANDATORY naming convention)
   - **MUST** contain exactly ONE state-changing call (to the unclamped handler)
   - **MUST** call unclamped handler, NOT the contract directly
   - Use `+ 1` in all modulo operations (MANDATORY)
   - No require/conditional logic
2. **Verify shortcut functions**:
   - **CRITICAL**: Confirm function is added ONLY to `TargetFunctions` contract (strict requirement)
   - Match an execution path from the `paths` array
   - Have descriptive name based on path conditions (e.g., `shortcut_{func}_byParam_condition1_condition2`)
   - Use clamped handlers for prerequisites
   - Use unclamped handler for target (when passing exact values)
   - Have 2+ state-changing calls

## Complete Examples

### Example 1: Clamped Handlers for Authorization

Multiple authorization patterns require different clamping approaches. Here are common scenarios:

#### Example 1a: Sender Authorization Check

**Coverage Gap:** Lines after `require(_isSenderAuthorized(onBehalf))` in `Vault::borrow` are uncovered
**Path Analysis:** Checking `merged-paths-prerequisites.json` shows this function has no prerequisites
**Decision:** **Clamped handler** - only needs valid parameter for `onBehalf`

```solidity
function vault_borrow_clamped(MarketParams memory marketParams, uint256 assets, uint256 shares, address receiver) public {
    // Clamp onBehalf to authorized actor to pass authorization check
    vault_borrow(marketParams, assets, shares, _getActor(), receiver);
}

function vault_borrow(MarketParams memory marketParams, uint256 assets, uint256 shares, address onBehalf, address receiver) public asActor {
    vault.borrow(marketParams, assets, shares, onBehalf, receiver);
}
```

#### Example 1b: Approval/Allowance Check

**Coverage Gap:** Lines after `require(allowance[from][msg.sender] >= amount)` in `Token::transferFrom` are uncovered
**Path Analysis:** Function has no prerequisites, but needs approved address pairs
**Decision:** **Clamped handler** - clamp `from` to address that has approved `_getActor()`

```solidity
function token_transferFrom_clamped(uint256 amount) public {
    // Clamp 'from' address to one that has approved the current actor
    // Assumes setup() established approval between actors
    address from = actors[0]; // Actor 0 has pre-approved other actors
    address to = actors[1];

    amount %= (token.balanceOf(from) + 1); // CRITICAL: + 1 is mandatory

    token_transferFrom(from, to, amount);
}

function token_transferFrom(address from, address to, uint256 amount) public asActor {
    token.transferFrom(from, to, amount);
}
```

#### Example 1c: Ownership Check

**Coverage Gap:** Lines after `require(ownerOf(tokenId) == msg.sender)` in `NFT::transferFrom` are uncovered
**Path Analysis:** Function has no prerequisites, but needs valid tokenId owned by actor
**Decision:** **Clamped handler** - clamp `tokenId` to one owned by `_getActor()`

```solidity
function nft_transfer_clamped(address to) public {
    // Get a tokenId owned by the current actor
    uint256 ownedTokenCount = nft.balanceOf(_getActor());

    // Clamp to a valid owned token index
    uint256 index = 0; // Could also clamp: index %= (ownedTokenCount + 1)
    uint256 tokenId = nft.tokenOfOwnerByIndex(_getActor(), index);

    nft_transfer(to, tokenId);
}

function nft_transfer(address to, uint256 tokenId) public asActor {
    nft.transferFrom(_getActor(), to, tokenId);
}
```

### Example 2: Shortcut Function for Conditional Branch

**Pattern:** Conditional branch not taken (Pattern 2)
**Coverage Gap:** Lines inside a conditional block checking state equality are uncovered (e.g., `if (collateral == 0)`, `if (balance == 0)`, `if (shares == totalShares)`)
**Path Analysis:**
- Target function: `vault_liquidate` (example - applies to any function with similar conditional)
- Execution path: "seizedAssets > 0 && position[id][borrower].collateral == 0 && data.length == 0"
- Prerequisites: Multiple calls needed to set up exact state (`vault_supply`, `vault_supplyCollateral`, `vault_borrow`, `oracle_setPrice`)
**Decision:** **Shortcut function** - requires multi-step sequence to create precise system state

```solidity
// Path: "seizedAssets > 0 && position[id][borrower].collateral == 0 && data.length == 0"
function shortcut_liquidate_bySeizedAssets_badDebt_noCallback(
    uint256 supplyAmount,
    uint256 collateralAmount,
    uint256 borrowAmount,
    uint256 newPrice
) public {
    // Prerequisites - use CLAMPED handlers
    vault_supply_clamped(supplyAmount);
    switchActor(1);
    vault_supplyCollateral_clamped(collateralAmount);
    vault_borrow_clamped(borrowAmount);

    // Make position unhealthy by changing oracle price
    oracle_setPrice(newPrice);

    // Read exact value to trigger "collateral == 0" branch
    Position memory pos = vault.position(defaultMarketId, _getActor());
    uint256 seizedAssets = pos.collateral; // Exact value to zero out collateral

    // Target call - use UNCLAMPED handler to preserve exact value
    switchActor(0);
    vault_liquidate(defaultMarketParams, _getActor(), seizedAssets, 0, "");
}

function vault_liquidate(MarketParams memory marketParams, address borrower, uint256 seizedAssets, uint256 repaidShares, bytes calldata data) public asActor {
    vault.liquidate(marketParams, borrower, seizedAssets, repaidShares, data);
}
```

### Example 3: Handler for Unreachable State

**Pattern:** Unreachable state (Pattern 3)
**Coverage Gap:** Lines after validation checks that fail due to uninitialized state (e.g., `require(price != 0)`, `require(initialized)`, `require(exists)`)
**Function Sequence Analysis:**
- State validation failing - system state is invalid/uninitialized
- No existing handler to initialize the required state
- Requires state-changing call before target functions can proceed
**Decision:** **New target handler** (if simple) or **Shortcut** (if complex setup)

**Example 3a: Simple State Initialization (New Handler)**

When state can be set with a single call:

```solidity
function oracle_setPrice_clamped(uint256 price) public {
    // Clamp token to known collateral token (using meaningful value)
    address token = address(collateralToken);

    oracle_setPrice(token, price);
}

function oracle_setPrice(address token, uint256 price) public {
    oracle.setPrice(token, price);
}
```

**Example 3b: Complex State Setup (Shortcut)**

When state requires multiple steps (e.g., initializing pool requires token approval + liquidity):

```solidity
function shortcut_swap_withInitializedPool(
    uint256 amount0,
    uint256 amount1,
    uint256 swapAmount
) public {
    // Initialize pool (multiple prerequisites)
    token0_approve_clamped(address(pool), amount0);
    token1_approve_clamped(address(pool), amount1);
    pool_initialize_clamped(amount0, amount1);

    // Now swap can proceed past require(pool.initialized)
    pool_swap(swapAmount);
}
```

These handlers enable the fuzzer to:
1. Initialize required system state (oracle prices, pool setup, etc.)
2. Allow target functions to proceed past state validation checks
3. Cover previously unreachable code paths

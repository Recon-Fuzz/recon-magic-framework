# Coverage Scoring Tool

A Python tool that enriches coverage analysis results with cyclomatic complexity scores from Slither and sorts functions by complexity.

## Purpose

This tool integrates static analysis (cyclomatic complexity from Slither) with dynamic analysis (coverage data from Echidna fuzzing) to help prioritize testing efforts. More complex functions with missing coverage should generally be prioritized for targeted fuzzing or manual testing.

## Installation

From the repository root:

```bash
pip install -e tools/covg_scoring/
```

## Usage

```bash
covg-scoring [-v] [--return-json]
```

The tool automatically looks for the `magic/` directory in your current working directory.

### Arguments

- `-v, --verbose`: Enable verbose output
- `--return-json`: Return JSON output to stdout instead of writing to file (useful for workflows)

### Examples

```bash
# Run the scoring tool (from the project root)
# It will automatically generate cyclomatic complexity data if not found
covg-scoring -v

# Or use --return-json to output to stdout (useful in workflows)
covg-scoring --return-json > magic/functions-missing-covg-sorted-1234567890.json
```

**Note:** The tool automatically generates cyclomatic complexity data using Slither's Python API if `magic/cyclomatic-complexity.json` doesn't exist. The complexity data is cached for future runs.

## What it does

1. Checks for existing cyclomatic complexity data in `magic/cyclomatic-complexity.json`
2. If not found, automatically generates it using Slither's Python API and caches it
3. Reads the most recent `functions-missing-covg-{timestamp}.json` file generated by `covg-eval`
4. Matches functions by contract name and function name
5. Adds a `complexity` field to each function entry
6. Sorts functions by complexity (highest first), with functions lacking complexity data at the end
7. Outputs the enriched and sorted data (either to file or stdout)

## Output Format

Each function entry in the output JSON will have an additional `complexity` field:

```json
{
  "function": "_adjustTrove",
  "contract": "BorrowerOperations",
  "source_file": "/path/to/BorrowerOperations.sol",
  "function_range": {
    "start": 100,
    "end": 150
  },
  "uncovered_code": {
    "line_range": "120-125",
    "last_covered_line": 119,
    "code": ["..."]
  },
  "complexity": 12
}
```

If no complexity data is found for a function, the `complexity` field will be `null`.

## Integration with Workflow

This tool is designed to be used in the `workflow-fuzzing-coverage` workflow after Step 14 (Evaluate Coverage):

```bash
# Step 14: Evaluate Coverage
covg-eval magic/ echidna/

# Step 15: Add complexity scores and sort
# (automatically generates complexity data if needed)
covg-scoring
```

## Cyclomatic Complexity

Cyclomatic complexity is a software metric that measures the number of linearly independent paths through a function's source code. Higher complexity indicates:

- More decision points (if/else, loops, switch statements)
- Higher testing difficulty
- Greater likelihood of bugs
- More edge cases to consider

By prioritizing complex functions with missing coverage, you can focus fuzzing efforts where they're most needed.
